## Приклад №1 Crash handler із SA_SIGINFO і register dump
### Код програми знаходиться в файлі `crash_diag.c`
```
nano crash_diag.c
```
У Linux, коли програма робить щось заборонене (наприклад, звертається до чужої пам'яті або ділить на нуль), процесор повідомляє про це операційну систему. Тоді ОС відправляє програмі спеціальний сигнал смерті (найчастіше це `SIGSEGV — Segmentation fault`). За замовчуванням програма просто вмирає і мовчки закривається.

Функція `install_crash_handlers` змінює цю поведінку.
У функції main програма спочатку налаштовує перехоплення, друкує свій `PID` (ідентифікатор), а потім викликає `crash_here()`

### Компіляція та запуск
```
gcc -Wall -Wextra -O0 -g -fno-omit-frame-pointer -no-pie crash_diag.c -o crash_diag
```
```
./crash_diag
```
Шукаєм рядок за RIP:
```
addr2line -e ./crash_diag -f -C 0x40171f
```

## Приклад №2 Correct sleeping: nanosleep() restart loop і clock_nanosleep() absolute deadline
### Код програми знаходиться в файлі `sleep_correct.c`
```
nano sleep_correct.c
```
Стандартні функції сну (наприклад, `sleep()` або `nanosleep())` перериваються, якщо програма отримує будь-який сигнал. Програма "прокидається" завчасно, і якщо це не обробити, розклад її роботи зламається.
На початку програми створюється обробник для сигналу `SIGUSR1`. Якщо хтось з іншого терміналу надсилає цей сигнал, викликається функція `on_usr1`, яка просто ставить прапорець `got_usr1 = 1`.
Зверніть увагу: змінна має тип `volatile sig_atomic_t`. Це гарантує, що вона безпечно зміниться навіть під час непередбачуваного переривання програми.
Друга частина програми демонструє періодичні "тіки" (наприклад, програма має щось робити рівно кожну секунду).
Якби ми використовували відносний сон (`sleep(1)`), виникла б проблема: виконання коду між снами забирає мілісекунди. Сон на 1 сек + робота 0.05 сек = цикл займає 1.05 сек. Через 20 циклів програма "відстане" на цілу секунду. Це називається `drift`. 

### Компіляція та запуск
```
gcc -Wall -Wextra -O2 sleep_correct.c -o sleep_correct
```
```
./sleep_correct
```

А в іншому терміналі треба написати 
```
kill -USR1 <pid>
```

## Приклад №3 Малий publisher-subscriber на real-time signals, sigqueue(), sigwaitinfo() і sigtimedwait()
### Код програми знаходиться в файлі `rt_pubsub.c`
```
nano rt_pubsub.c
```

Зазвичай ми звикли, що сигнали (наприклад, `SIGINT` при натисканні `Ctrl+C або SIGKILL`) використовуються просто як "стусан" для програми — вони не несуть ніякої додаткової інформації.
Але ця програма використовує `Real-time` сигнали (починаючи з `SIGRTMIN`), які мають дві суперздібності.
Функція `subscriber` запускається в першому терміналі і перетворює програму на "поштову скриньку".
Якщо запущено варіант `sub-timeout`, використовується `sigtimedwait` з таймером на 5 секунд. Якщо за 5 секунд ніхто нічого не написав, вона видає повідомлення `timeout: no messages...` і чекає знову.

Функція publisher запускається в другому терміналі. Її задача — взяти числа, які ви написали в командному рядку, і "вистрілити" ними у підписника.
Замість стандартної команди kill (яка просто шле голий сигнал), тут використовується sigqueue. Вона дозволяє "прикріпити" до сигналу структуру `union sigval`, в яку програма попередньо записує наше число (`value.sival_int`).
Операційна система бере цей сигнал разом із числом і кладе його в чергу до процесу-підписника.

### Компіляція та запуск
```
gcc -Wall -Wextra -O2 rt_pubsub.c -o rt_pubsub
```
Запуск у першому терміналі
```
./rt_pubsub sub
```

Запуск у другому терміналі
```
./rt_pubsub pub <PID_SUBSCRIBER> 10 20 30 -1
```

Варіант із timeout:
```
./rt_pubsub sub-timeout
```

## Завдання варіант 7
**Реалізуйте послідовне породження процесів, де кожен створений процес створює ще один дочірній, усього п’ять разів. Прослідкуйте порядок завершення.**

Головна фішка цієї програми полягає в тому, що цикл продовжує виконувати лише новостворена дитина. Батьківський процес завжди зупиняється.
* Виклик `fork()`
Програма просить операційну систему створити свою точну копію. Тепер у нас є два процеси: старий (батько) і новий (нащадок).

* Блок `else`
Оскільки для `батька pid > 0`, він потрапляє у блок `else`. Там він одразу зустрічає команду `wait(NULL)`. Коли дитина нарешті вмирає, батько "прокидається", друкує своє повідомлення `<<< Process ... waited ...` і відразу викликає `exit(EXIT_SUCCESS)`. Тобто батько ніколи не йде на наступну ітерацію циклу. Його життєвий шлях закінчується тут.

* Блок `else if (pid == 0))`
Дитина потрапляє у цей блок, друкує повідомлення `Step X: created child process...`.Якщо це не остання ітерація, дитина доходить до кінця тіла циклу і... йде на наступну ітерацію (i++). На наступному кроці циклу ця вчорашня дитина сама викликає `fork()` і стає батьком для нового процесу.

### Код програми знаходиться в файлі `chain_fork.c`
```
nano chain_fork.c
```

### Компіляція та запуск
```
gcc -Wall -Wextra chain_fork.c -o chain_fork
```
```
./chain_fork
```

Головний процес створює Нащадка 1 і "засинає" (через `wait`), очікуючи на його смерть. Нащадок 1 бачить, що він не останній, створює Нащадка 2 і теж "засинає". Це триває доти, поки не народиться Нащадок 5. Саме завдяки комбінації `fork()`, `wait()` та `exit()` створюється ідеально рівний ланцюжок, який потім послідовно "згортається" з кінця до початку.
