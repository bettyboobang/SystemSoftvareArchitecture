# Практична робота №11
## Індивідуальне завдання(7 варіант)
Реалізуйте сигналобезпечну функцію обробки помилки, яка намагається коректно завершити роботу з усіма ресурсами, включаючи сокети, тимчасові файли та потокові буфери.

Насправді ми не можемо безпечно працювати з потоковими буферами (наприклад, `FILE *`, `fflush`, `fclose`, `printf`) всередині обробника сигналів. 
Функції стандартної бібліотеки вводу-виводу використовують внутрішні блокування та динамічну пам'ять. 
Якщо сигнал перерве програму в момент, коли вона працювала з `printf` або `malloc`, виклик `fflush` або `fclose` в обробнику призведе до `deadlock` або `segmentation fault`

Тому реалізація повинна ділитися на те, що можна безпечно робити в обробнику, і те, для чого потрібен інший підхід.

### Що є безпечним для обробника сигналів?
* Закриття файлових дескрипторів сокетів та файлів
* Видалення тимчасових файлів
* Вивід тексту через системні виклики
* Екстрене завершення

Щоб коректно скинути потокові буфери, найкращою практикою є встановлення атомарного прапорця (`volatile sig_atomic_t`) в обробнику сигналів. 
Основний цикл програми перевіряє цей прапорець і безпечно завершує роботу (включаючи `fflush`)

### Код програми знаходиться в файлі `server_cleanup.c`
Створіть файл програми, для початку
```
nano server_cleanup.c
```

### Компіляція та запуск
Скомпілюйте програму звичним способом
```
gcc -o server_cleanup server_cleanup.c
```
```
./server_cleanup
```

Для того щоб надіслати `SIGINT` потрібно натиснути комбінацію клавіш `Ctrl+C`.

Можна побачити, що в результаті спочатку програма запустилася і вивела повідомлення про успішне створення сокета та тимчасового файлу. 
Далі на екрані з'являється символ `^C`, який свідчить про те, що ви натиснули комбінацію клавіш `Ctrl+C`, надіславши програмі сигнал на завершення.
Замість того, щоб раптово перервати роботу, програма перехопила цей сигнал і запустила підготовлену функцію очищення. 
Наступні рядки демонструють послідовне і безпечне вивільнення ресурсів: програма успішно закрила сокет, потім закрила дескриптор тимчасового файлу і відразу ж видалила цей файл з системи.
Також на екрані з'явилося попередження про те, що буфери виводу не були скинуті задля уникнення взаємного блокування. 
Як доказ цього, ми не бачимо на екрані того тестового рядка, який ми передавали у `printf` без символу нового рядка в кінці - програма навмисно відкинула ці дані, щоб гарантувати стабільність обробника сигналів. 
Наприкінці виводиться повідомлення про безпечне завершення процесу через системний виклик `_exit`, після чого керування успішно повертається до терміналу.
