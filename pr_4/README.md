# Практична робота №4

## Завдання 1
**Скільки пам’яті може виділити malloc(3) за один виклик? Параметр malloc(3) є цілим числом типу даних size_t, тому логічно максимальне число, яке можна передати як параметр malloc(3), — це максимальне значення size_t на платформі (sizeof(size_t)). У 64-бітній Linux size_t становить 8 байтів, тобто 8 * 8 = 64 біти. Відповідно, максимальний обсяг пам’яті, який може бути виділений за один виклик malloc(3), дорівнює 2^64. Спробуйте запустити код на x86_64 та x86. Чому теоретично максимальний обсяг складає 8 ексабайт, а не 16?**

### Чому 8 ексабайт, а не 16?
Параметр функції `malloc(3)` має тип `size_t`. 
У 64-бітній системі його максимальне значення становить 2^64 - 1 байтів (приблизно 16 ексабайт).
Однак теоретичний ліміт виділення пам'яті становить рівно половину — **8 ексабайт**. 

На це є дві причини:
* Арифметика вказівників. У мові C результат віднімання двох вказівників має знаковий тип `ptrdiff_t`. Щоб уникнути переповнення під час обчислення відстані між елементами в пам'яті, стандарт і бібліотека C обмежують максимальний розмір одного об'єкта до максимального додатного значення, тобто 2^63 - 1;
* Розподіл віртуального адресного простору. У сучасних ОС з віртуальною пам'яттю загальний адресний простір ділиться навпіл. Нижня половина доступна для програм користувача, а верхня жорстко зарезервована для ядра операційної системи. Тому програма має доступ лише до половини адресного простору.

**Код для виконання даного завдання знвходиться в `malloc.c`**

Створимо програму, що намагається виділити 16 ЕБ (максимум для `size_t` та 8 ЕБ пам'яті, щоб перевірити поведінку системи та компілятора
```
nano malloc.c
```

### Компіляція та запуск

**Скомпілюйте код командою:**
```
gcc -Wall malloc.c -o malloc
```
Компілятор видасть попередження `warning: argument 1 value exceeds maximum object size`. Він попереджає про стандарт С і попереджає що значення `SIZE_MAX` перевищує безпечний ліміт для вказівників

**Далі запустіть код командою:**
```
./malloc.c
```

### Що ми бачимо після запуску x86_64 програми?
* Переповнення `int`. Математичний результат 3 000 000 000 не помістився у 32-бітний знаковий `int`.
Відбулося переповнення, і значення стало від'ємним: -1 294 967 296.
* Приведення до `size_t`. На 64-бітній системі тип `size_t` має розмір 64 біти.
Коли ми прередажмо туди від'ємне 32-бітне число, процесор спочатку розширює його до 64 біт зі збереженням знака, а потім інтерпретує як беззнакове.
В результаті ми отримуємо гігантське число 18 446 744 072 414 584 320 _(це понад 18 екзабайтів)._

**Висновок**. Система фізично не має такого обсягу пам'яті, тому `malloc` миттєво повертає `NULL`.

### Що ми бачимо після запуску x86 програми?

Одразу можна побачити що поведінка програми кардинально змінилась етапі виклику `malloc`

* Переповнення `int`. Відбулося точно так само, значення `num` дорівнює -1 294 967 296.
* Приведення до `size_t`. На 32-бітній архітектурі тип `size_t` є 32-бітним.
Тому розширення розміру не відбувається. Комп'ютер просто бере бітове представлення від'ємного числа -1 294 967 296 і починає читати його як беззнакове.
Прикол в тому, що бітове представлення цього від'ємного числа в точності дорівнює беззнаковому числу 3 000 000 000.

**Висновок**. У цьому випадку `malloc`спробував виділити 3 гігабайти пам'яті. 
Однак у 32-бітних системах адресний простір одного процесу зазвичай обмежений до 3 ГБ.
Знайти один суцільний блок пам'яті розміром 3 ГБ неможливо, тому операційна система відмовляє, і `malloc` також просто повертає `NULL`.

### Остаточний висновок
Це завдання показує важливість перевірки арифметичних операцій на переповнення перед викликом будь-яких функцій розподілу пам'яті.
В нашому завданні програма може і повернула `NULL`, але в багатьох інших випадках і обставинах коли преповнення призврдить, наприклад, до величезного дотадного числа це стає причиною вразливостей _Heap Buffer Overflow._

### Що ми можем з цього зрозуміти?
Отриманий вивід дозволяє зробити ключові висновки щодо менеджере пам'яті в `glibc`
* `malloc(0)` не повертає `NULL`. Незважаючи на те, що ми просимо 0 байтів, `glibc` виділяє мінімальний блок пам'яті і повертає валідну адресу (у мене 0x5eebc13b82a0).
Це зроблено для зворотної сумісності зі старим кодом та уникнення помилок `Segmentation Fault`;
* Вказівник, отриманий від `malloc(0)`, містить правильні приховані метадані .
Тому виклик `free()` успішно розпізнає цей блок і звільняє його без жодних помилок;
* Виклик `malloc(3)` повертає точно таку ж адресу, як і `malloc(0)`. Це відбувається через те, що:
  1. І для 0, і для 3 байтів `glibc` виділяє блок однакового мінімального розміру (зазвичай 24-32 байти);
  2. Після виклику `free()`пам'ять не віддається операційній системі миттєво. Вона поміщає цей блок у швидкий кеш;
  3. Наступний виклик `malloc(3)` знаходить цей вільний блок ідеального розміру в кеші та миттєво перевикористовує його, заощаджуючи ресурси системи.

  ## Завдання 4
  **Чи є помилки у такому коді?**
  ``` c
  void *ptr = NULL;
   while (<some-condition-is-true>) {
    if (!ptr)
        ptr = malloc(n);
   // [... <використання 'ptr'> ...]
       free(ptr);
  }
  ```

 **Напишіть тестовий випадок, який продемонструє проблему та правильний варіант коду.**

### В чому проблема?
У початковому коді була допущена логічна помилка в циклі:

```c
void *ptr = NULL;
while (<some-condition-is-true>) {
    if (!ptr)
        ptr = malloc(n);
    
    // ... використання ptr ...
    
    free(ptr);
    //Тут помилка вказівник ptr не скидається у NULL після звільнення пам'яті
}
```
Функція `free(ptr)` звільняє виділену пам'ять, але не змінює адресу, що зберігається у змінній ptr. 
На наступній ітерації циклу умова if (!ptr) не виконується, пам'ять не виділяється повторно, і програма намагається записати дані у звільнену ділянку пам'яті, а потім вдруге її звільнити. 
Це призводить до невизначеної поведінки або аварійного завершення програми
