## Завдання 1
**Скільки пам’яті може виділити malloc(3) за один виклик? Параметр malloc(3) є цілим числом типу даних size_t, тому логічно максимальне число, яке можна передати як параметр malloc(3), — це максимальне значення size_t на платформі (sizeof(size_t)). У 64-бітній Linux size_t становить 8 байтів, тобто 8 * 8 = 64 біти. Відповідно, максимальний обсяг пам’яті, який може бути виділений за один виклик malloc(3), дорівнює 2^64. Спробуйте запустити код на x86_64 та x86. Чому теоретично максимальний обсяг складає 8 ексабайт, а не 16?**

### Чому 8 ексабайт, а не 16?
Параметр функції `malloc(3)` має тип `size_t`. 
У 64-бітній системі його максимальне значення становить 2^64 - 1 байтів (приблизно 16 ексабайт).
Однак теоретичний ліміт виділення пам'яті становить рівно половину — **8 ексабайт**. 

На це є дві причини:
* Арифметика вказівників. У мові C результат віднімання двох вказівників має знаковий тип `ptrdiff_t`. Щоб уникнути переповнення під час обчислення відстані між елементами в пам'яті, стандарт і бібліотека C обмежують максимальний розмір одного об'єкта до максимального додатного значення, тобто 2^63 - 1;
* Розподіл віртуального адресного простору. У сучасних ОС з віртуальною пам'яттю загальний адресний простір ділиться навпіл. Нижня половина доступна для програм користувача, а верхня жорстко зарезервована для ядра операційної системи. Тому програма має доступ лише до половини адресного простору.

**Код для виконання даного завдання знвходиться в `malloc.c`**

Створимо програму, що намагається виділити 16 ЕБ (максимум для `size_t` та 8 ЕБ пам'яті, щоб перевірити поведінку системи та компілятора
```

```

### Що ми можем з цього зрозуміти?
Отриманий вивід дозволяє зробити ключові висновки щодо менеджере пам'яті в `glibc`
* `malloc(0)` не повертає `NULL`. Незважаючи на те, що ми просимо 0 байтів, `glibc` виділяє мінімальний блок пам'яті і повертає валідну адресу (у мене 0x5eebc13b82a0).
Це зроблено для зворотної сумісності зі старим кодом та уникнення помилок `Segmentation Fault`;
* Вказівник, отриманий від `malloc(0)`, містить правильні приховані метадані .
Тому виклик `free()` успішно розпізнає цей блок і звільняє його без жодних помилок;
* Виклик `malloc(3)` повертає точно таку ж адресу, як і `malloc(0)`. Це відбувається через те, що:
  1. І для 0, і для 3 байтів `glibc` виділяє блок однакового мінімального розміру (зазвичай 24-32 байти);
  2. Після виклику `free()`пам'ять не віддається операційній системі миттєво. Вона поміщає цей блок у швидкий кеш;
  3. Наступний виклик `malloc(3)` знаходить цей вільний блок ідеального розміру в кеші та миттєво перевикористовує його, заощаджуючи ресурси системи.

  **Чи є помилки у такому коді?**
  ``` c
  void *ptr = NULL;
   while (<some-condition-is-true>) {
    if (!ptr)
        ptr = malloc(n);
   // [... <використання 'ptr'> ...]

  }
  ```

 **Напишіть тестовий випадок, який продемонструє проблему та правильний варіант коду.**

### В чому проблема?
У початковому коді була допущена логічна помилка в циклі:

```c
void *ptr = NULL;
while (<some-condition-is-true>) {
    if (!ptr)
        ptr = malloc(n);
    
    // ... використання ptr ...
    
    free(ptr);
    //Тут помилка вказівник ptr не скидається у NULL після звільнення пам'яті
}
```
Функція `free(ptr)` звільняє виділену пам'ять, але не змінює адресу, що зберігається у змінній ptr. 
На наступній ітерації циклу умова if (!ptr) не виконується, пам'ять не виділяється повторно, і програма намагається записати дані у звільнену ділянку пам'яті, а потім вдруге її звільнити. 
Це призводить до невизначеної поведінки або аварійного завершення програми

1. **Ітерація 1.** `ptr` був `NULL`, тому `malloc` успішно виділив пам'ять. Дані записалися, пам'ять звільнилася через `free(ptr)`.

Робота зі звільненою пам'яттю викликає невизначену поведінку. 
В залежності від версії ОС, бібліотеки `glibc` та розміру виділеної пам'яті, менеджер пам'яті може іноді дозволити подвійне звільнення маленького блоку без миттєвого краху.
Проте у реальному великому проєкті така помилка призведе до пошкодження даних, вразливостей або раптового падіння програми в зовсім іншому місці.

## Завдання 6
**Якщо realloc(3) викликати з NULL або розміром 0, що станеться? Напишіть тестовий випадок.**

Програма перевіряє три конкретні сценарії роботи функції. 
У першому випадку, якщо викликати `realloc(NULL, size)`, функція поведе себе абсолютно аналогічно до `malloc(size)`. Вона виділить новий блок пам'яті вказаного розміру і поверне вказівник на нього.
У другому випадку, при виклику `realloc(ptr, 0)`, поведінка буде аналогічною до `free(ptr).` 
Це означає, що функція звільнить раніше виділену пам'ять за вказаною адресою та поверне `NULL `.
Третій випадок це виклик `realloc(NULL, 0)`. Є еквівалентним виклику `malloc(0)`.
Результатом такого виклику стане повернення унікального вказівника на спеціальний фіктивний блок пам'яті "нульового розміру". 
Важливо пам'ятати, що до цього блоку не можна звертатися для читання чи запису, але його обов'язково потрібно звільнити за допомогою `free() `, щоб уникнути витоків пам'яті.

### Висновок
Використання `reallocarray` є сучасною та безпечною практикою в програмуванні на C. 
На відміну від класичного `realloc`, `reallocarray` бере на себе відповідальність за безпечне обчислення необхідного обсягу пам'яті, унеможливлюючи виникнення критичних вразливостей типу `Integer Overflow` під час роботи з динамічними масивами.

| Характеристика | `realloc` | `reallocarray` |
| :--- | :--- | :--- |
| **Сигнатура** | `realloc(ptr, 2000)` | `reallocarray(ptr, 500, 4, ...)` |
| **Передача розміру** | Один аргумент | Два окремі аргументи |
| **Місце обчислення** | Обчислення розміру відбувається в коді користувача до виклику функції| Перемноження делегується функції всередині системної бібліотеки |
| **Захист від переповнення** |**Відсутній.** Якщо результат множення перевищить ліміт `size_t`, станеться непомітне переповнення |**Присутній.** Функція перевіряє можливість цілочисельного переповнення перед множенням|

### А яка причина таких результатів?
**Кроки 1, 4, 16 байтів**

Процесор завантажує дані з оперативної пам'яті в надшвидкий кеш L1 не по одному байту, а блоками кеш-лініями, які на більшості сучасних архітектур мають розмір 64 байти. При малих кроках ми звертаємося до даних, які вже були завантажені в кеш під час попередньої ітерації.

**Крок 64 байти**

Оскільки розмір кеш-лінії дорівнює 64 байтам, кожен такий стрибок змушує процесор звертатися до нової кеш-лінії. Це спричиняє постійні промахи кешу. Процесор змушений простоювати, поки дані підтягуються з повільніших рівнів L2/L3 або безпосередньо з RAM.

**Крок 4096 байтів**

4096 байтів (4 КБ) стандартний розмір сторінки віртуальної пам'яті в Linux. При такому кроці кожне звернення відбувається до нової сторінки пам'яті. 
Процесор має конвертувати віртуальну адресу у фізичну за допомогою кешу адрес TLB (Translation Lookaside Buffer).
Оскільки ми постійно стрибаємо по нових сторінках, TLB не може вмістити їх усі, виникають часті промахи (TLB Misses), і системі доводиться робити апаратний обхід таблиць сторінок  в оперативній пам'яті, що є дуже дорогою операцією.

### Помилки що виникли під час виконання завдання

1. Використання маленької літери у прапорці `-o3` призвело до помилки. Компілятор сприйняв це як вказівку назвати вихідний файл цифрою 3, тоді як для увімкнення максимального рівня оптимізації, який є критично важливим для нашого бенчмарку, необхідно використовувати велику літеру, тобто `-O3`;

2. Наступна складність виникла на етапі препроцесингу через синтаксис макросів.
Під час оголошення константи розміру масиву запис без пробілу перед дужкою змусив компілятор інтерпретувати це як макрос-функцію, яка очікує передачі аргументів, а не як звичайну заміну тексту.
Додавання пробілу між назвою макросу та його значенням одразу вирішило проблему нерозпізнаних параметрів;

3. І останнє це помилка з використанням високоточної функції вимірювання часу вимагало підключення бібліотеки реального часу за допомогою додаткового прапорця лінкера `-lrt`.
