# Практична робота №5
## Варіант 7
### Завдання
**Зробити запис за межі локального масиву так, щоб перезаписати іншу локальну змінну без краху.**

В цій практичній роботі треба було вивчити нову інформацію щодо типових проблем при роботі з пам'яттю.
Для виконання даного завдання, звичайна компіляція не підійде варто використати певні хитрощі.

### Але спочатку **створимо файл** для коду цього завдання
```
nano local_matrix.c
```
Вхдідний код цієї програми знаходиться у файлі `local_matrix.c`

**В написаному коді ми:**
1. Спочакту оголошуємо локальні змінні;
2. Робимо запис за межі масиву;
3. І індекс 3 вказуватиме на комірку пам'ятв де лежить змінна target.

### Специфічна компіляція та запуск

Сучасні компілятори дуже розумні і за замовчуванням намагаються захистити нас від помилок. 
Вони можуть переставити змінні місцями у пам'яті або додати спеціальний "захист стеку" (`stack canary`), який помітить втручання і примусово зупинить програму ( і видасть помилку `stack smashing detected`).

Якщо ми спробуємо скомпілювати цю програму звичною нам командою
```
gcc -Wall local_matrix.c -o matrix
```
Ми побачимо ось такий результат:
<img width="751" height="94" alt="Знімок екрана 2026-03-05 114712" src="https://github.com/user-attachments/assets/54817dae-3991-49cb-9809-8a213bc7f55a" />

Він нас не влаштовує, тож щоб гарантовано відтворити перезапис в терміналі треба виконати ось цю команду:
```
gcc -O0 -fno-stack-protector -g local_matrix.c -o matrix
```
### Шо означає ця команда?
`-O0` Повністю вимикає оптимізацію. 
Це змушує компілятор покласти змінні `target` і `buffer` у стек рівно в тому порядку, в якому ми їх оголосили, і не видаляти наш "неправильний" рядок коду.

`-fno-stack-protector` Вимикає захист стеку. 
Завдяки цьому компілятор не розставляє "пастки" навколо масиву, і ми можемо вільно вилізти за його межі, перезаписавши сусідню пам'ять.

`-g` Додає налагоджувальну інформацію (щоб Valgrind міг показати номери рядків, якщо ми захочемо його перевірити)

Тож наступним кроком буде запуск команди
```
./matrix
```

## Як це працює і як програма не викликає крах?
_Розташування у стеку_

Локальні змінні зберігаються у стеку впритул одна до одної. 
Коли ми звертаємося до елемента за межами масиву (наприклад, до 4-го елемента масиву з 3-х), ми фактично записуємо дані за адресою пам'яті, яка йде одразу за масивом.

_Відсутність  `segfault`_

Аварійне завершення (segfault) виникає тоді, коли програма намагається отримати доступ до забороненої пам'яті або пам'яті, яка їй не виділена. 
Оскільки змінна `target` також знаходиться у нашому легальному фреймі стеку, операційна система не бачить порушення прав доступу і не "вбиває" процес

_Невизначена поведінка_

Варто пам'ятати, що наслідки таких операцій не визначені стандартом. 
Те, яка саме змінна буде перезаписана, сильно залежить від компілятора та його оптимізацій (він може переставити змінні місцями або додати між ними порожнє місце для вирівнювання). 
Як і у випадку з іншими невизначеностями, результат може відрізнятися на різних компіляторах. 
Щоб побачити ефект гарантовано, під час компіляції варто вимикати оптимізації, про які було написано раніше

### Аналіз результатів
  Змінні зазвичай розміщуються там послідовно. 
Оскільки ми використовували прапорці для вимкнення захисту та оптимізацій, змінна `target` та масив `buffer` розташовуються в пам'яті впритул одне до одного.
Спочатку програма запам'ятовує, що змінна `target` має значення 5. 

  Далі вона створює масив buffer на три елементи. 
У мові C нумерація індексів починається з нуля, тому ці три елементи - це `buffer[0], buffer[1] та buffer[2]`.

  Коли ми просимо програму надрукувати `target` уперше, вона звертається до комірки пам'яті, виділеної для цієї змінної, бачить там число 5 і виводить його на екран.
Далі відбувається ключовий момент помилки. Програма бачить "запиши число 99 у buffer[3]". 

  Мова C не перевіряє розміри масиву під час виконання.
Це класичний випадок виходу за межі - звернення до пам'яті поза масивом.
Щоб виконати команду, програма бере адресу початку масиву, відступає від неї на відстань трьох елементів і записує дані.

  Оскільки елемента з індексом 3 у нашому масиві не існує, ця нова адреса вказує точно на те місце в пам'яті, де одразу за масивом зберігається змінна `target`. Внаслідок цього виникає пошкодження пам'яті, тобто запис потайки змінює пам'ять іншої змінної. 
Ну і програма записує число 99 поверх старої п'ятірки.

  Саме тому, коли ми вдруге просимо надрукувати `target`, програма йде до тієї ж самої комірки, але тепер знаходить там 99. Краху (`segfault`) не стається, бо ми записали дані в дозволену для нашої програми область пам'яті стеку, просто схитрувавши з адресацією.
