## Практична робота №6

### Завдання (Варіант 7)

**Написати програму, що активно використовує heap, потоки й mmap,
після чого проаналізувати її адресний простір через /proc/self/maps**

Програма повинна виконувати такі дії
* Виділення пам'яті на купі
Використовує функцію `malloc` для динамічного виділення пам'яті

* Створення потоків
Запускає додаткові потоки за допомогою бібліотеки `pthread`.
У Linux стеки для нових потоків створюються за допомогою системного виклику `mmap`.

* Пряме використання mmap
Використовує системний виклик `mmap` для виділення анонімної ділянки пам'яті великого розміру.

* Самоаналіз адресного простору
Програма читає та виводить у консоль вміст системного файлу `/proc/self/maps`, що дозволяє наочно побачити, де саме в пам'яті розташовані код програми, купа, `mmap`-ділянки та стеки потоків.

* Коректне звільнення ресурсів
Після завершення роботи програма звільняє всю виділену пам'ять (`free`, `munmap`) та коректно завершує потоки (`pthread_join`).

### Створіть файл heap.c який знаходиться в іншому файлі цієї папки
```
nano heap.c
```

### Компіляція та запуск
Для компіляції програми необхідно підключити бібліотеку `pthread` це стандартна бібліотека для створення та керування потоками всередині однієї програми

```
gcc -g -pthread heap.c -o heap
```

Ну а далі просто запустіть програму
```
./heap
```

### Розберемо результати запуску програми

Під час запуску програма виведе свій PID та таблицю мапінгу пам'яті.

**Спочатку ми бачимо виконуваний код**
```
5d8599564000-5d8599569000 ... /home/bettyboobang/SystemSoftvareArchitecture/pr_6/heap
```
Це сегменти скомпільованої програми. 
Ділянка з правами `r-xp (read, execute)` містить безпосередньо машинний код програми, а `rw-p `- глобальні змінні.

**Далі купа**
```
5d85b4ed8000-5d85b4ef9000 rw-p 00000000 00:00 0 [heap]
```
Тут знаходиться пам'ять, яку ми динамічно виділили за допомогою функції `malloc()` у головному потоці. 
Права доступу `rw-p` дозволяють читати та писати туди дані.

**Потім бачимо ділянку `mmap`**
```
7fee2f600000-7fee30a00000 rw-p 00000000 00:00 0
```
Цей великий безіменний блок з правами `rw-p` - це пам'ять, яку ми виділили прямим системним викликом `mmap`. 
Якщо відняти адреси (0x30a00000 - 0x2f600000), ми отримаємо рівно 0x1400000 шістнадцяткових байтів, що дорівнює рівно 20 МБ.

**Після цього видно стеки створених потоків**
```
7fee30bfe000-7fee313fe000 rw-p 00000000 00:00 0
7fee313ff000-7fee31bff000 rw-p 00000000 00:00 0
```
Кожен додатковий потік потребує власного стеку. 
Бібліотека `pthread` створює їх також за допомогою `mmap`. В результатах чітко видно ці стеки, кожен розміром по 8 МБ.

_Варто знати_. Перед кожним таким стеком йде сторінка пам'яті з правами `---p`. Це так звана захисна сторінка. Якщо потік переповнить свій стек, він наткнеться на цю сторінку, і програма впаде з помилкою `Segmentation fault`, замість того, щоб непомітно пошкодити пам'ять іншого потоку.

**Далі бачимо розділювані бібліотеки**
```
7fee32600000-7fee32805000 ... /usr/lib/x86_64-linux-gnu/libc.so.6
```
Це підключена стандартна бібліотека мови C (`glibc`), яка завантажена в пам'ять.

**І останні строчки це стек головного процесу та системні ділянки**
```
7fffa03f0000-7fffa0411000 rw-p 00000000 00:00 0 [stack]
... [vvar], [vdso], [vsyscall]
```
`[stack]` це стек основного потоку програми, він знаходиться на найвищих адресах. 
Блоки `[vdso]` та `[vsyscall]`  це спеціальні оптимізації ядра лінукс, які "проєціюють" частину системних функцій прямо в пам'ять нашої програми, щоб системні виклики  працювали супер швидко.

### Додатково перевіримо наявність витоків пам'яті та помилок доступу за допомогою Valgrind
Якщо до цього ви не мали Valgrind встановіть його на віртуальну машину
```
sudo apt update && sudo apt install valgrind -y
```
Так, команди update i install можна виконувати разом за допомогою `&&`

**Тепер можна запустити програму під контролем valgrind**
```
valgrind --tool=memcheck --leak-check=full --show-leak-kinds=all ./heap
```
